Codex 反复写出或读出乱码的原因说明

这不是“Codex 情绪不稳定”，本质上是编码边界不一致导致的结果，尤其是在 Windows + PowerShell + 混合工具链的环境里更容易反复出现。主要原因有下面几类：

1. 终端显示编码和文件实际编码不一致
很多前端文件本身是 UTF-8，但 PowerShell 默认输出编码、控制台代码页、某些命令的读取方式不一定是 UTF-8。这样文件内容本来是对的，终端里却会显示成乱码；如果再把终端里已经乱码的文本复制回文件，就会把“显示错误”变成“真实文件错误”。

2. 文件曾经被按错误编码读过一次，又按另一种编码写回
最常见的是 UTF-8 内容被当成 GBK、ANSI 或其他本地编码读取，读出来变成“锟斤拷”或一串异常字符；随后再保存，就会把原始字节信息破坏掉。之后模型再读取这个文件时，拿到的已经不是原文，而是被污染过的文本。

3. 局部修补发生在已经损坏的文本附近
当一个文件里同时存在正常中文和错误字节时，模型在做局部补丁时很容易“沿用附近的坏文本模式”。这会表现成：明明只改一两个按钮文案，但相邻标题、注释或占位文案又重新出现乱码。

4. 工具链对同一文件的处理方式不统一
编辑器、Shell、测试工具、打包工具、Git diff 查看方式，如果没有统一到 UTF-8，会出现“编辑器里正常、终端里乱码”或者“终端里正常、页面里乱码”的错位现象。模型在这种不一致环境里工作，最容易被误导。

5. 模型会继承它刚刚看到的上下文
如果模型读取到的源文件已经带乱码，它继续补代码时，往往会把那段乱码当成“现有风格”的一部分延续下去。这不是模型在主动制造乱码，而是在错误上下文里做了连续补全。

这次在待办中心里出现的情况，基本就是以上几种因素叠加：
- 文件里本来已经混入了一部分乱码文本；
- 终端读取时又放大了乱码观感；
- 后续局部修改沿着坏文本继续补，导致页面上一些中文正常、一些中文继续坏掉。

后续避免同类问题，建议固定这几条：

1. 所有前端源码统一保存为 UTF-8。
2. 在 Windows 终端里读取中文文件时显式使用 UTF-8。
3. 不要把终端里已经显示成乱码的文本直接复制回源码。
4. 遇到局部乱码较多的文件，优先整段重写为干净 UTF-8，而不是在坏字节附近反复缝补。
5. 提交前用页面实际显示、测试和构建结果一起确认，而不是只看终端输出。

简短结论：
Codex 反复读写乱码，通常不是“反复无常”，而是编码环境、历史坏字节和局部补丁三件事叠加后的连锁反应。统一 UTF-8、避免复制乱码上下文、必要时整段重写，是最稳的解决办法。
