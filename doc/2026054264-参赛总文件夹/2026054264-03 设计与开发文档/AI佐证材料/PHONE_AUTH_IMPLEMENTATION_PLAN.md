# 手机号优先多入口认证改造方案

## 1. 目标定义

本次改造的目标不是把数据库主键从 `uid` 改成手机号，而是把“业务主登录标识”和“默认注册入口”切换为手机号，同时保留现有邮箱体系作为可选通道。

建议的最终体验如下：

1. 注册
   - 首选：手机号 + 图形验证码 + 短信验证码注册。
   - 支持无密码快捷注册。
   - 仍保留邮箱 + 用户名 + 密码注册。
2. 登录
   - 支持手机号 + 验证码登录。
   - 支持邮箱 + 密码登录。
   - 支持用户名 + 密码登录。
3. 找回密码
   - 如果绑定了手机号，则可通过手机号验证码重置密码。
   - 如果绑定了邮箱，则可通过邮箱验证码或邮件链接重置密码。
   - 有什么通道就提供什么通道。
4. 绑定与换绑
   - 在设置页或个人页增加“绑定手机号 / 绑定邮箱”入口。
   - 绑定、换绑都走验证码或邮件验证流程。

## 2. 关键决策

### 2.1 保留 `uid` 作为数据库主键

不建议把手机号作为数据库主键。原因如下：

1. 当前系统的真实主键是 `uid`，所有关联表都依赖它。
2. JWT、管理员接口、历史记录、收藏、待办、政策发布、民生反馈等链路都以 `user_id` 为核心。
3. 把手机号改成数据库主键的收益极低，但会显著放大迁移成本和数据风险。

本方案将“手机号主键”解释为“手机号成为主认证标识和默认注册方式”。

### 2.2 保留邮箱体系，不做一刀切删除

邮箱体系继续保留，用于：

1. 邮箱 + 密码登录。
2. 邮箱注册与验证。
3. 忘记密码的邮箱找回。
4. 权限变更、升级申请等通知邮件。

这符合“主打一个有什么就提供什么”的要求，也能兼容现有用户。

### 2.3 不直接复用当前 `User.phone` 作为登录手机号

当前种子数据中的 `User.phone` 混有 `010-12345001` 这类固话/联系号码，不适合直接当短信登录手机号。

推荐方案：

1. 保留现有 `phone` 的业务语义，视为联系号码或历史兼容字段。
2. 新增一个明确用于认证的字段，例如：
   - `login_phone`
   - 或 `mobile_phone`
3. 前端对用户仍统一展示为“绑定手机号”，但后端不要直接拿旧 `phone` 字段承载短信登录能力。

这是本次改造中最重要的兼容性决策之一。

## 3. 最终产品形态

## 3.1 登录页

登录页建议改为两个主入口：

1. `手机号验证码登录`
   - 手机号
   - 图形验证码
   - 短信验证码
   - 登录按钮
2. `密码登录`
   - 输入框标题直接写成“邮箱或用户名”
   - 密码输入框
   - 忘记密码入口

说明：

1. 邮箱 + 密码登录和用户名 + 密码登录可以共用一个密码登录表单，不需要拆成两个 tab。
2. 后端根据输入内容判定是邮箱还是用户名，或者由前端显式传 `identity_type`。

## 3.2 注册页

注册页建议改为两个主入口：

1. `手机号快捷注册`
   - 手机号
   - 图形验证码
   - 短信验证码
   - 用户名或昵称
   - 可选密码
2. `邮箱注册`
   - 邮箱
   - 用户名
   - 密码
   - 确认密码
   - 邮箱验证码或邮件验证链接

说明：

1. 虽然手机号注册可以无密码，但仍建议在注册阶段收集 `用户名/昵称`，因为当前系统大量页面直接展示 `uname`。
2. 手机号快捷注册表单里可以提供一个“同时设置密码”的可选开关，而不是强制要求密码。

## 3.3 忘记密码

忘记密码建议做成两步：

1. 用户输入账号标识
   - 手机号
   - 邮箱
   - 用户名
2. 后端返回可用找回方式
   - 已绑定手机号：`138****0001`
   - 已绑定邮箱：`z***@example.com`

然后用户选择一种方式继续：

1. 通过手机号验证码重置密码。
2. 通过邮箱验证码重置密码。
3. 通过邮件链接跳转重置密码页面。

这样既符合“有什么就提供什么”，也能避免在第一步过度暴露账户信息。

## 3.4 个人中心 / 设置页

在个人页和设置页增加以下账号安全卡片：

1. 绑定手机号
   - 已绑定 / 未绑定
   - 已验证 / 未验证
   - 绑定
   - 换绑
2. 绑定邮箱
   - 已绑定 / 未绑定
   - 已验证 / 未验证
   - 绑定
   - 换绑
3. 登录密码
   - 已设置密码 / 未设置密码
   - 修改密码
   - 设置密码

如果账户是“手机号快捷注册且未设置密码”，则页面优先展示“设置密码”，而不是“修改密码”。

## 4. 数据模型建议

## 4.1 `User` 模型

推荐保留 `uid`，并对 `User` 模型扩展以下字段：

1. 保留
   - `uid`
   - `uname`
   - `email`
   - `email_verified`
   - `hashed_pwd`
   - `role`
2. 新增
   - `login_phone` 或 `mobile_phone`
   - `phone_verified`
   - `password_login_enabled`
   - `preferred_login_method`
   - `last_login_method`

说明：

1. `password_login_enabled` 用来表达该用户是否启用了密码登录。
2. 对于“无密码手机号注册”用户，`hashed_pwd` 可以为空，或者使用专用哨兵值，但更推荐允许为空并显式使用 `password_login_enabled`。
3. `uname` 仍然建议保持唯一。
4. `email` 建议改成可选，但如果有值则仍应唯一。
5. `login_phone` 建议改成可选，但如果有值则必须唯一。

## 4.2 联系号码与登录手机号分离

如果系统后续仍需要展示机构联系电话、固定电话或对外联系方式，建议明确区分：

1. `phone` 或 `contact_phone`
   - 联系方式
   - 允许固话
2. `login_phone`
   - 认证手机号
   - 只允许标准手机号码

这是避免历史数据污染认证体系的关键。

## 4.3 验证码与挑战状态

不建议把短信验证码、图形验证码直接塞到 `User` 表中。

推荐建立独立的认证挑战存储：

1. 图形验证码 challenge
2. 手机验证码 challenge
3. 邮箱验证码 challenge
4. 密码重置 challenge
5. 绑定手机号 / 绑定邮箱 challenge

建议的通用字段：

1. `challenge_id`
2. `channel`
   - `captcha`
   - `sms`
   - `email`
3. `purpose`
   - `register`
   - `login`
   - `reset_password`
   - `bind_phone`
   - `bind_email`
4. `target`
   - 手机号或邮箱
5. `code_hash`
6. `expires_at`
7. `used_at`
8. `attempt_count`
9. `request_ip`
10. `extra_json`

### 4.3.1 存储实现建议

建议优先采用：

1. Redis
2. Redis 不可用时回退到内存

原因：

1. 当前项目已经有 Redis 与内存双栈回退基础。
2. 验证码天然属于短生命周期、可失效、可一次性消费的数据。
3. Phase 1 无需为此单独落库。

如果后续部署为多实例，再考虑把 challenge 管理迁移到更严格的 Redis-only 或独立表结构。

## 5. 接口设计建议

建议新增独立的 `/auth` 路由组，不把所有逻辑继续堆到现有 `/login/` 和 `/user` 下。

## 5.1 图形验证码

1. `GET /auth/captcha`
   - 返回 `captcha_id`
   - 返回 `svg` 或 base64 图像
2. `POST /auth/captcha/verify`
   - 输入 `captcha_id` + `answer`
   - 返回一个短期 `captcha_pass_token`

也可以不单独做 `/verify`，而是在发送短信 / 邮件验证码时直接携带图形验证码答案。但独立 `captcha_pass_token` 结构更清晰，也更便于风控。

## 5.2 手机号验证码

1. `POST /auth/phone/send-code`
   - 用途：登录、注册、重置密码、绑定手机号
   - 请求：手机号、用途、图形验证码通过凭证
   - 返回：发送成功、过期时间、冷却时间
   - 在沙箱环境下额外返回 `preview_code`
2. `POST /auth/phone/register`
   - 请求：手机号、验证码、用户名、可选密码
3. `POST /auth/phone/login`
   - 请求：手机号、验证码
4. `POST /auth/phone/bind`
   - 绑定或换绑手机号

## 5.3 邮箱体系

1. `POST /auth/email/register`
   - 请求：邮箱、用户名、密码
2. `POST /auth/email/send-code`
   - 用途：注册验证、重置密码、绑定邮箱
3. `POST /auth/email/verify-code`
4. `GET /auth/email/verify-link`
5. `POST /auth/email/bind`

说明：

1. 当前已有邮件预览与真实邮件能力，可以复用。
2. 邮箱验证仍建议保留“验证码 + 邮件链接”双能力。

## 5.4 密码登录

1. `POST /auth/login/password`
   - 输入：`identity`、`password`
   - `identity` 可为邮箱或用户名
   - 后端自动判定，或显式传 `identity_type`

兼容策略：

1. 现有 `/login/` 可以暂时保留。
2. `/login/` 内部逐步改造成对 `/auth/login/password` 的兼容包装。

## 5.5 忘记密码

1. `POST /auth/password/recovery/options`
   - 输入：手机号、邮箱或用户名
   - 输出：可用找回通道的脱敏列表
2. `POST /auth/password/recovery/send`
   - 按选定通道下发验证码或邮件
3. `POST /auth/password/reset`
   - 输入：challenge、验证码、两次新密码

## 5.6 设置密码 / 修改密码

1. `POST /auth/password/set`
   - 适用于无密码账户首次设置密码
2. `POST /auth/password/change`
   - 适用于已有密码用户修改密码

## 6. 前端改造建议

## 6.1 认证弹窗与登录注册页

现有统一认证入口已经收敛到 Home 组件体系，这一点应继续保留。

建议改造方向：

1. `LoginForm.vue`
   - 增加模式切换：
     - 手机号验证码登录
     - 密码登录
2. `RegisterForm.vue`
   - 增加模式切换：
     - 手机号快捷注册
     - 邮箱注册
3. 新增找回密码面板
   - 可以作为弹窗内第三个视图
   - 也可以单独做 `ForgotPasswordForm`

## 6.2 登录默认策略

建议默认显示“手机号验证码登录”。

原因：

1. 符合“手机号优先”的目标。
2. 对普通用户更轻。
3. 与沙箱短信预览配合后，演示效果更直接。

## 6.3 密码登录表单

建议文案统一为：

1. 标签：`邮箱或用户名`
2. 占位：`请输入邮箱或用户名`

不建议再分成“邮箱密码登录”和“用户名密码登录”两个独立 tab，因为交互收益不大，维护成本更高。

## 6.4 手机号快捷注册表单

建议字段：

1. 手机号
2. 图形验证码
3. 短信验证码
4. 用户名/昵称
5. 可选密码

建议新增说明文案：

1. 不设置密码也可注册，后续可在设置页补充。
2. 当前为沙箱短信，验证码会在界面中展示。

## 6.5 设置页与个人页

建议增加如下 UI 项：

1. `绑定手机号`
2. `换绑手机号`
3. `绑定邮箱`
4. `换绑邮箱`
5. `设置密码`
6. `修改密码`

同时在用户资料区展示：

1. 手机号绑定状态
2. 邮箱绑定状态
3. 密码状态
4. 最近登录方式

## 6.6 管理员页

管理员用户表建议补充：

1. 登录手机号
2. 手机号验证状态
3. 邮箱验证状态
4. 密码启用状态
5. 最近登录方式

## 7. 风控与安全设计

## 7.1 图形验证码

图形验证码必须由后端生成并校验，不能只做前端假验证码。

推荐实现：

1. 后端生成简单 SVG 验证码。
2. 验证答案只存 challenge，不回传明文。
3. 验证通过后签发短期 `captcha_pass_token`。

## 7.2 频率限制

项目已经有 `rate_limit_service.py`，建议直接复用。

至少需要以下限流桶：

1. `auth:captcha:create:ip`
2. `auth:captcha:verify:ip`
3. `auth:sms:send:ip`
4. `auth:sms:send:phone`
5. `auth:sms:verify:phone`
6. `auth:email:send:ip`
7. `auth:email:send:email`
8. `auth:password:recovery:ip`

推荐阈值：

1. 同一手机号发送验证码冷却 60 秒。
2. 同一手机号 10 分钟内最多发送 5 次。
3. 同一 IP 10 分钟内最多发送 10 次短信验证码。
4. 同一 challenge 最多尝试 5 次。
5. 图形验证码 5 分钟过期。
6. 短信/邮箱验证码 5 到 10 分钟过期。

## 7.3 验证码存储与校验

必须遵循：

1. 只存哈希，不存明文。
2. 一次性使用。
3. 成功后立即失效。
4. 超过尝试次数即作废。

## 7.4 沙箱短信策略

为了兼顾演示与真实感，建议：

1. 后端发送前 `sleep 0.8 ~ 1.5s`。
2. 仅在开发/沙箱模式返回 `preview_code`。
3. 生产环境绝不返回验证码明文。

前端可以在验证码输入区附近展示：

1. `沙箱验证码：123456`
2. 或单独的“开发预览框”

## 7.5 换绑安全

换绑手机号或邮箱时，不建议只靠当前登录态直接修改。

推荐至少满足以下之一：

1. 验证旧绑定通道。
2. 输入当前密码。
3. 管理员人工处理高风险换绑。

如果旧通道不可用，再走“账号恢复流程”。

## 8. 兼容与迁移策略

## 8.1 老用户兼容

已有用户不应被强制迁移。

兼容策略：

1. 已有邮箱 + 密码用户继续可登录。
2. 已有用户名 + 密码用户继续可登录。
3. 未绑定手机号的老用户可在设置页后续绑定。
4. 已有 `email_verified` 状态全部保留。

## 8.2 种子数据兼容

由于现有 `phone` 含固话，不建议直接把老字段视为“已绑定短信手机号”。

建议处理方式：

1. 旧 `phone` 保留为联系信息。
2. 新增 `login_phone` 为空。
3. 只有用户主动绑定并验证后，才视为可用于短信登录的手机号。

## 8.3 无密码账户兼容

手机号快捷注册允许无密码，但要明确以下规则：

1. 无密码账户仍可正常用手机号验证码登录。
2. 无密码账户默认不能走密码登录。
3. 设置密码后，才启用邮箱/用户名 + 密码登录能力。

## 8.4 用户名规则

因为系统大量使用 `uname` 展示用户身份，建议：

1. 手机号注册时仍要求输入用户名/昵称。
2. 若后续想进一步降低门槛，可自动生成默认用户名，但这会带来重名、脏数据和后续补填问题。

当前阶段不建议省略 `uname`。

## 9. 实施顺序

3. 增加短信验证码发送与校验。
4. 增加统一 challenge 管理。

## Phase 2：手机号优先登录注册

4. 保留原有邮箱注册与密码登录。

## Phase 3：找回密码与账号安全

1. 新增找回密码流程。

## 11. 本方案的推荐结论

推荐采用以下最终路线：

1. `uid` 不动，继续做数据库主键。
2. 手机号成为默认注册入口和默认登录入口。
3. 邮箱体系保留为可选认证和恢复通道。
4. 密码登录统一成“邮箱或用户名 + 密码”。
5. 手机号注册允许无密码，但必须保留 `uname`。
6. 由于旧 `phone` 字段被固话污染，认证手机号应使用新字段，不直接复用旧字段。
7. 图形验证码和频率限制必须在 Phase 1 一起落地，不能后补。

## 12. 仍需确认的产品细节

以下问题建议在正式开工前确认：
