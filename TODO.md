- [ ] 实现Agent的记忆:
对于像你这样正在开发 **ClearNotify** 的开发者来说，理解 Agent 的记忆（Memory）至关重要。

简单来说，Agent 的记忆并不是像人类大脑那样通过神经元存储，而是通过**“外部存储 + 动态注入”**的方式实现的。

目前的实现方案主要分为以下三个层级：

1. 短期记忆 (Context Window)
这是最基础的，本质上就是**对话历史的堆叠**。
* **实现原理：** 将用户之前的提问和 AI 之前的回答（Chat History）全部重新塞进 Prompt 里发给模型。
* **局限性：** 模型的 Context Window（如 128k tokens）是有上限的。对话越长，成本越高，且早期的信息会被“挤掉”。

2. 长期记忆 (RAG / Vector Database)
这是目前主流的“记忆存取”方案，也是你在 **拾知** 项目中可能会用到的（结合 PostgreSQL 或 Redis）。
* **实现原理：** 1.  **存储：** 将对话内容或知识点通过 Embedding 模型转化为**向量（Vector）**，存入向量数据库。
    2.  **检索：** 当用户提问时，Agent 先计算问题的向量，在数据库里寻找最相似的 Top-K 条记录。
    3.  **注入：** 把找回来的旧记忆作为“参考资料”塞进当前的 Prompt 中。
* **技术栈：** OpenAI Embedding + ChromaDB / PGVector (PostgreSQL)。

3. 结构化记忆 (Entity Memory / Knowledge Graph)
对于复杂的 Agent，仅仅靠向量搜索是不够的（因为向量搜索有时会“断章取义”）。
* **实现原理：** Agent 具备一个额外的**“反思步（Reflection）”**。在对话结束后，Agent 会自动提取关键信息：
    * “用户提到他的队友使用 MacOS。”
    * “用户目前的开发框架是 FastAPI。”
* **存储：** 这些结构化信息会被存入 JSON、数据库或图数据库中。下次对话时，系统会自动加载这些“用户画像”信息。

记忆管理的常见架构比较:

| 记忆类型 | 存储介质 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **缓冲记忆 (Buffer)** | 内存/Redis | 极快，保持对话连贯。 | 容易忘事（受限于长度）。 |
| **摘要记忆 (Summary)** | 数据库 | 节省 Token，保留核心意图。 | 会丢失技术细节（比如特定的 Bug 描述）。 |
| **向量记忆 (Vector)** | Vector DB | 海量存储，支持语义检索。 | 检索可能不精准，且缺乏时间线感。 |


开发建议:
既然你正在处理 **ClearNotify** 这种 AI 驱动的工具，建议你可以尝试以下组合：

1.  **Redis 存储近期 Session：** 保证当前解析任务的连贯。
2.  **PostgreSQL (PGVector) 存储长期知识：** 用于存储用户过去解析过的公告模式或偏好。
3.  **MCP 联动：** 如果你写了一个 MCP Server，你的 Agent 可以直接通过 MCP 接口去读取本地文件或数据库里的历史记录，这在某种程度上就是给 Agent 装了一个“可插拔的记忆硬盘”。
